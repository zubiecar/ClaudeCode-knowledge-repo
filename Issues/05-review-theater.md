## Review Theater: When Code Review Becomes a Formality

**Related to:** [Issues Overview](overview.md) — Issue 5

---

### What Is Review Theater?

Review theater is the condition in which the code review process continues to operate — PRs are opened, comments are left, approvals are given — but without the substantive human scrutiny that makes review valuable. The form persists; the function disappears. It is the gap between a team that "does code review" and a team whose code review actually catches problems.

In AI-assisted development, review theater emerges from a specific convergence of pressures: PRs are growing larger, AI-generated code is arriving faster than reviewers can absorb it, and an implicit social logic suggests that code generated by Claude or Copilot is "probably fine." The result is that the last human checkpoint before code enters the codebase is increasingly a formality — a checkbox completed under time pressure by a reviewer who is not reading carefully enough to catch what the AI got wrong.

---

### The Scale of the Problem

The DORA 2025 report, drawing on nearly 5,000 developer responses, documents the mechanical consequence of AI adoption on review workload: a **154% increase in pull request size** and a **91% increase in code review time** — without a corresponding increase in organizational delivery performance.[^3] Teams are spending nearly twice as long reviewing PRs that are more than twice as large, while producing software of roughly equivalent quality to the pre-AI baseline.

The LogRocket analysis of real AI-generated code provides concrete measurements: a REST endpoint generated by AI ran to 186 lines vs. 29 lines for a manually written equivalent (6.4×). An error-handling refactor: 288 lines vs. 26 (1,100% larger). Reviewing AI code took 2–4x longer — not because of more bugs found, but because reviewers must evaluate whether defensive patterns are justified, infer the AI's intent from its output, and verify correctness without being able to rely on familiarity with the patterns used.[^6]

Meanwhile, Faros AI's telemetry from 10,000+ developers across 1,255 teams found that teams using AI merge **98% more PRs** while review time rises 91%. The rate of PRs created has more than doubled; the rate at which any single PR can receive genuine attention has declined proportionally.[^7 from prompt fragmentation research, adapted]

### The Automation Complacency Mechanism

The psychological underpinning of review theater is well-documented in cognitive science: automation bias — the tendency to favor and insufficiently scrutinize suggestions from automated systems.

A 2025 systematic review of 35 peer-reviewed studies across cognitive psychology, HCI, and neuroscience found that automation bias is specifically amplified by the conditions of velocity-driven development: **high workload, time pressure, and task complexity**.[^7] When engineers are reviewing under sprint pressure, the cognitive shortcut of trusting AI output is most likely to be engaged. Critically, even explainability features — when present — can reinforce misplaced trust when explanations are too technical or too simple to be actionable.

A 2025/2026 paper on trust in AI coding assistants found that software engineering research "rarely defines or conceptualizes trust," instead conflating it with mere acceptance of AI-generated content. This means the field is systematically underestimating inappropriate reliance: teams and researchers reporting "high trust in AI tools" may actually be reporting automation complacency rather than calibrated confidence.[^8]

The practical manifestation: an arXiv study (January 2026) analyzed reviewer sentiment vs. code quality in AI-generated PRs. It found that AI PRs exhibit significantly higher code redundancy and lower reuse of existing modules than human-written PRs — yet sentiment analysis shows reviewers express **more positive or neutral reactions to AI contributions despite the quality deficit**.[^5] The surface-level plausibility of well-formatted, syntactically clean AI code sedates reviewers precisely when they should be most vigilant.

---

### PR Size and Review Quality

The relationship between PR size and review depth is empirically established.

LinearB's 2025 analysis of 6.1 million pull requests across 3,000 teams found that PR size is "the single most significant driver of velocity across the entire PR lifecycle," and that review time scales **non-linearly with PR size** — meaning the 154% increase in PR size documented by DORA does not produce 154% more review attention, but something closer to 91% more time producing diminishing quality per line reviewed.[^3b]

Cognitive research confirms the mechanism: a 2026 empirical study (Lund University / Empirical Software Engineering) built a cognitive model of code review from think-aloud studies with 10 professional developers across 34 real reviews. Review is a two-phase decision process — orientation → analysis — where the orientation phase is highly sensitive to context deprivation. Large, opaque AI-generated PRs that don't map clearly to familiar patterns short-circuit the analytical phase; reviewers form incomplete mental models and approve code they have not genuinely understood.[^1]

### When AI Reviews AI: The Theater Enablement Trap

A tempting response to the volume problem is to deploy AI-assisted code review tools — CodeRabbit, Copilot code review, Qodo — to handle the scale that human reviewers cannot. The empirical picture on these tools is mixed at best.

A large-scale study of 22,000+ AI review comments across 178 repositories and 16 different tools found that AI review comments are most likely to produce actual code changes only when they are concise, include code snippets, and are manually triggered at the hunk level. **Most deployed AI review configurations produce feedback that developers ignore** — the appearance of review without the substance.[^12]

A subsequent arXiv study (April 2026) tested the industry claim that code review agents can autonomously handle 80% of PRs. The empirical finding: code review agent-only PRs achieved a **45.20% merge rate vs. 68.37% for human-only reviews** — a 23-point gap. 60.2% of closed agent-only PRs had low signal ratios. The paper concludes CRAs should augment rather than replace human reviewers — and that teams deploying them as a primary review mechanism are operating with majority-noise review pipelines.[^13]

### The Social Pressure Dynamic

Review theater is not only a cognitive problem. It is also a cultural one.

Stack Overflow's February 2026 survey of developer sentiment found that **84% use or plan to use AI tools**, but trust dropped to 29% (down 11 percentage points year-over-year). More developers actively **distrust** AI accuracy (46%) than trust it (33%). Experienced developers distrust most — with the highest "highly distrust" rate at 20%.[^11]

This creates a revealing tension: most engineers know they should not fully trust AI-generated code, but most are operating in organizations where that code is flowing through review faster than it can be genuinely scrutinized. The engineer who blocks a PR for quality reasons on a team that is measuring velocity and shipping frequently does so against an implicit cultural current. Without explicit leadership endorsement that slowing down for quality is a success rather than a failure, the social incentives favor rubber-stamp approval.

---

### Proposed Solutions

**1. Enforce PR Size Limits for AI-Generated Code**

No AI-generated PR should exceed 300–400 lines. Larger AI outputs must be split before review. This is not a quality preference — it is a necessary condition for genuine review. LinearB's data shows review quality degrades non-linearly with size; above 400 lines, the probability of meaningful review approaches baseline chance.[^3b] Add a CI check that flags oversized PRs before review begins, requiring the author to split the PR.

**2. Separate Architectural Review from Implementation Review**

These require different reviewers and different cognitive modes. Architectural review — does this fit our system, does it respect module boundaries, does it introduce new abstractions appropriately — should be conducted by the architect or a senior engineer with full-system context. Implementation review — does this work correctly, are there bugs, are edge cases handled — is appropriate for a more junior reviewer. Conflating them into a single review produces neither well.

Salesforce's engineering blog documents their production experience with this separation: as AI-driven code volume increased ~30%, they built a system to reconstruct developer intent semantically rather than by diff inspection — maintaining a "second-pair-of-eyes guarantee" on architectural risk while delegating mechanical review to tooling.[^10]

**3. Require Authors to Explain AI-Generated PRs**

PR descriptions for AI-generated code should include a plain-language explanation of what the code does and why it is structured the way it is — not what the AI generated, but what the author understands about it. If the author cannot write this description, the PR should not be merged. This is the review-side complement to the Explanation Gate described in the Comprehension Debt memo, and it serves double duty: it forces author comprehension before review begins, and it gives reviewers a mental model to evaluate rather than inferring intent from unfamiliar code.

**4. Create Cultural Permission to Block**

The head of the team should explicitly and verbally name the norm: a blocked PR for quality reasons is a success, not a failure. This framing must come from leadership and be repeated at retrospectives. Without it, the implicit velocity pressure overrides the individual engineer's judgment about when to push back.

**5. Use AI Review Tools as Supplements, Not Replacements**

AI code review tools add value when they are configured to catch specific, well-defined vulnerability patterns (see Issue 4) and when their outputs are treated as input to human review rather than as a substitute for it. The AppSec Santa study found that hybrid SAST+LLM achieves 89.5% precision on security findings. But the arXiv CRA study found that autonomous AI review produces a 45.20% merge rate with high noise ratios. The distinction is: AI tools as specialized scanners, yes; AI tools as the primary reviewer, no.

---

### Summary

Review theater is what happens when the volume and format of AI-generated code outpaces the team's capacity to review it genuinely. It is not a failure of individual diligence — it is a structural mismatch between how much AI is producing and how much cognitive bandwidth human reviewers have. The correction is structural: smaller PRs that are reviewable, separated review modes that match cognitive requirements to reviewer expertise, explicit author comprehension requirements, and cultural norms that give reviewers permission to slow down. Without these, code review continues to exist as a process while ceasing to function as a checkpoint.
